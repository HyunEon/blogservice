# 🌟 BlogService: Django 기반 풀스택 블로그 프로젝트

- 데모: https://blog.hyuneon.org/

## 💡 프로젝트 개요
- Django를 활용하여 제작된 "블로그 커뮤니티" 프로젝트입니다. 
  - Django의 개발 과정과 웹 앱 구조를 이해하기 위한 학습 프로젝트이기도 합니다.
  - 처음 구상 시, 혼자 사용하려고 만들었던 프로젝트에서 다수의 사용자가 공통으로 사용할 수 있는 서비스는 어떻게 만들어야 하는지 궁금해서 지금의 블로그 커뮤니티 프로젝트까지 확장했습니다.
- 로그인, 회원가입, 본문 작성/수정/삭제, 댓글 작성/수정/삭제 등의 블로그 커뮤니티 기능을 제공합니다.

## ❓ 왜 이 프로젝트를 시작했는가?
- 개인적인 생각, 기록 등을 적고 싶어 시작했습니다. 제가 무언가 일상이나 생각 같은 것을 기록하는 것을 좋아하는 편이라 네이버 블로그나 일기 앱에 작성하곤 했는데, 문득 직접 만든 사이트에 적으면 정말 멋지겠다는 단순한 생각에서 출발했습니다.

- 여러 성격의 웹 서비스 중, 블로그로 시작한 것은 저를 포함한 많은 사람들이 오래전부터 사용해왔고 다양한 용도로 활용할 수 있는 것이 블로그라고 생각했기 때문입니다. 블로그는 사용자에 따라 다양한 성격의 사이트가 만들어질 수 있다고 생각했습니다. 

- 네이버 블로그, 티스토리 등의 서비스를 개인적인 용도 뿐만 아니라 비즈니스 성격을 가진 블로그를 운영하는 사용자 층도 매우 많다는 점을 고려했습니다.

## ❓ 무엇을 배웠는가?
- Django의 ORM, 인증, 미들웨어 등 기능 활용을 통해 웹 앱의 기본 구조를 이해하고 경험했습니다.
- Docker Compose를 활용한 다중 Docker 컨테이너 관리 및 배포를 경험했습니다.
- 비동기 작업을 구현해보며 비동기 환경의 기본 지식과 왜 필요한 지를 직접 경험했습니다.
- Nginx를 통해 리버스 프록시를 직접 구성해보며, 구조에 대해 이해했습니다.
- 배포 및 cloudflare에서 도메인을 직접 구입해서 호스팅까지 진행해보며, 실제 운용 가능한 웹 앱 구축을 경험했습니다.

## ⚙️ 기술 스택
|분류|사용된 기술|설명|
|------|---|---|
|언어|Python|블로그 서비스의 전반적인 구현을 위해 필요한 언어|
|프론트엔드|HTML, CSS, JS, Jquery(일부)|사용자에게 표시될 블로그 UI를 구성하기 위한 기본 요소|
|백엔드|Django, Gunicorn|서버 로직과 웹 기능 구현을 위한 프레임워크 및 이를 운영하는 웹 서버|
|DB|PostgreSQL|영구적인 데이터를 저장하는 데 사용된 데이터베이스 시스템|
|캐시, 메시지 큐|Redis|비동기 작업을 위한 메시지 브로커 및 데이터 캐싱 도구|
|비동기 처리|Celery|비동기 백그라운드 작업 처리를 위한 도구|
|미디어 서버|MinIO|미디어 파일 저장을 위한 S3 호환 스토리지|
|인프라|Cloudflare Tunnel, Nginx|외부 접근 경로, 리버스 프록시 및 보안 설정에 관여하는 시스템|
|배포|Docker, Docker Compose|모든 서비스를 컨테이너화하고 관리하는 데 필요한 도구|
|프로젝트 관리|Git, Pipenv|프로젝트 디렉토리 및 형상 관리를 위해 사용|

## 🎯 왜 Django인가?
- Instargram, Pinterest 등 글로벌 기업에서 Django를 활용한 서비스를 제공했었다는 점에서 왜 그랬는지 궁금했었고, Django의 대표적인 특징인 빠른 서비스 빌드와, 웹 앱을 위한 수 많은 편리한 기능(미들웨어, 인증, ORM, 어드민 기능 등..) 및 강력한 보안 기능을 자체적으로 제공하여 개발자는 이를 선택하여 간편하게 기능을 구현할 수 있다는 점이 1인 프로젝트를 구상 중인 저에게는 매우 매력적으로 들렸습니다.

- 무엇보다 Python 특유의 직관적이고 쉬운 문법으로 풀 프레임 워크 웹 앱을 빌드할 수 있다는 점이 더욱 흡족했었습니다. 물론 Python을 사용하는 프레임 워크로 Flask도 존재하지만 전통적인 "CRUD 중심의 블로그 서비스"를 만들기에 제공하는 기능들이 너무 적었고, 프로젝트 특성 상, 인증 중심의 CRUD가 많은 서비스는 Django가 압도적으로 많은 기능을 지원하고 있어 제외했습니다.

- 무언가 퀄리티 있게 상호 작용이 많은 웹 앱을 만들기보다는, 전통적인 웹 앱 구조에 대해 학습하고 어떤 방식으로 돌아가는지 경험하고 공부하는 것이 목적이었습니다. 그러므로 개발 난이도도 비교적 단순한 SSR 기반의 Django가 적절하다고 판단하여 최종적으로 선택하게 되었습니다.

## 😟 아쉬웠던 점
### 운영 환경의 아쉬움
- 클라우드 환경을 사용하기에 부담스러운 비용과 과금 요소들의 부담이 커, 블로그 프로젝트는 현재 제 개인 홈 서버인 M1 Mac Mini와 Cloudflare Tunnel을 사용하는 온프레미스 구조로 제작되었습니다. 그렇기에, 클라우드 환경에서의 DevOps를 경험하지 못하는 게 아쉬웠습니다.

### 높았던 AI 의존도
- LLM 덕분에 문제 해결은 빠르게 할 수 있었으나, 코드가 어떤 방식으로 동작하고 왜 이 구조로 짰을까를 이해하고 넘어가는 과정 속에서, 순수하게 코드를 직접 작성하기 보다는 "이런 로직으로 가면 될 것 같은데.." 하면서 GPT/Gemini 같은 LLM 에이전트를 활용해 코드를 작성했던 적이 생각보다 많았던 것 같습니다.

- 다음에 진행하는 프로젝트에서는 보다 더 많은 부분을 직접 설계하며 코드 레벨에서 더 깊이 이해하고 다시 활용하는 단계로 발전해가고 싶습니다.
